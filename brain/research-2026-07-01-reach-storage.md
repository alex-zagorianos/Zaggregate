# Research — Local storage architecture for a growing job corpus (2026-07-01)

_Confidence: high — grounded in current JobScout source (tracker/db.py, scrape/company_registry.py, app.spec, requirements.txt) plus 2025-2026 web sources for SQLite/DuckDB/vector-search facts; medium on exact vector-lib benchmark numbers (single third-party source)_

## Summary
Stay on SQLite — it is the correct engine for JobScout at every scale this app will realistically reach (100k inbox rows, 100k company-registry entries), and switching to DuckDB or a dedicated vector DB would add packaging risk for no benefit today. The concrete gaps are elsewhere: the WAL pragmas are incomplete (missing `synchronous=NORMAL`), there is no full-text search despite `description` already being stored, the company registry is a flat JSON file with no index that will start costing real time above ~10k entries, and there is no provenance/versioning model for multi-source ingestion. If/when semantic search on job descriptions is added, `sqlite-vec` (pure-C, ~160KB wheel, PyInstaller-friendly) or plain NumPy brute-force (JobScout's realistic embedding count is in the thousands-to-low-hundred-thousands, well within brute-force's practical range) are the only vector options that fit the free/local/single-file/no-server constraint — DuckDB and LanceDB are heavier, unnecessary, and (for DuckDB with extensions) carry documented Windows PyInstaller extension-loading bugs.

## Findings
## Current state (verified in repo)

- `tracker/db.py` runs one SQLite file per project (`projects/<slug>/tracker.db`), WAL mode, `busy_timeout=30000`, `SCHEMA_VERSION`-gated migrations with a `.bak-v<N>` copy before each ALTER. Tables: `applications`, `inbox` (dedup key = `norm_url` UNIQUE, computed by `models.normalize_url`), `dismissed`, `runs`, `status_history`, `score_history`, `contacts`. This is a solid, idiomatic SQLite schema for its current size — largest observed real file today is ~5.3MB (`projects/controls-cincinnati/tracker.db`), i.e. a few thousand rows.
- `get_conn()` ([tracker/db.py:55-63](tracker/db.py:55)) sets only `busy_timeout` and `journal_mode=WAL`. It does **not** set `synchronous=NORMAL`, `temp_store=MEMORY`, or `mmap_size` — the three other pragmas [phiresky's widely-cited SQLite tuning benchmark](https://phiresky.github.io/blog/2020/sqlite-performance-tuning/) uses to hit 100k SELECTs/sec at multi-GB scale with concurrent readers. SQLite's own WAL docs confirm `synchronous=NORMAL` in WAL mode is corruption-safe and only loses the last transaction on an OS crash/power-loss (never on an app crash) — it is the official recommendation for WAL-mode apps ([sqlite.org/wal.html](https://sqlite.org/wal.html), corroborated by [SQLite forum](https://sqlite.org/forum/info/d1a6bc7c4cd2baca)). For a GUI+daily-run app writing hundreds of inbox rows per run, this is a free, safe win.
- No FTS5 anywhere in the codebase (`grep -i fts5` across `*.py` returns nothing). `inbox.description` already stores up to 5000 chars of JD text ([tracker/db.py:528](tracker/db.py:528)) but there is no local full-text search over it — the GUI's only "search" is the upstream API keyword query (`gui.py:1890-2092`), not a query over what's already saved. As the inbox grows past a few thousand triaged rows, a user (especially dad, non-technical) has no way to find "that one posting that mentioned Epic" again.
- The company/board registry (`companies.json`, currently 301 real entries / 65KB — [discover/registry.py](discover/registry.py), [scrape/company_registry.py:155-215](scrape/company_registry.py:155)) is a flat JSON file, fully re-parsed with `json.loads(path.read_text())` on **every** call to `get_registry()`, `has_industry()`, `industry_company_count()`, `registry_stats()`, and rewritten whole-file on every `save_companies()`. At 301 entries this is instant (measured: 0.00s). At the dimension's stated target of 10k-100k boards, this becomes a real linear-scan-on-every-call cost with zero indexing by `ats_type`/`slug`/`industry`, and JSON has no built-in dedup/UNIQUE enforcement (`save_companies` does an O(n) Python set-membership check each call instead of a DB constraint).
- `coverage/registry_history.py` similarly persists estimator snapshots as JSON files per industry, not SQLite rows — same "no query surface, no index" limitation once history spans many runs/industries.
- No embeddings/vector storage exists yet; `coverage/estimators.py:45` lazy-imports `numpy` only for one GLM computation and it is **not** in `requirements.txt` today (confirmed: `pip`/`py -3.12 -c "import numpy"` fails locally — it's an optional soft dependency, not bundled).
- `app.spec` already proves the packaging pattern that matters here: it collects submodules per first-party package and treats `scrapling`'s heavy browser binaries as an opt-in downloaded-on-demand extra rather than a bundled dependency — i.e., JobScout's own convention is "stdlib/pure-Python/small-wheel by default, heavy binary deps only behind an explicit user opt-in." Any storage/vector choice should honor that same rule.

## SQLite vs DuckDB vs hybrid

SQLite remains the right primary store: it's OLTP-shaped (many small point writes/reads — exactly `inbox_add_many`, `update_job`, `inbox_set_fit`), it's Python-stdlib (zero packaging risk, already proven in the frozen exe), and it comfortably scales to multi-GB / 100k+ rows with the right pragmas — SQLite's own docs and third-party benchmarks put practical size limits in the hundreds-of-GB range for this workload class, far beyond anything JobScout will hit. DuckDB is architecturally the opposite: a columnar, vectorized OLAP engine that is 10-50x faster than SQLite for full-table aggregates (`AVG`, `GROUP BY` over millions of rows) but is not meant to replace a transactional store — the consensus 2026 guidance (MotherDuck, PostHog, Kestra, Better Stack) is "SQLite for the OLTP store, DuckDB attached read-only for the analytics layer," not a wholesale replacement. Concretely for JobScout: the `coverage/estimators.py` Chapman/Chao1/Good-Turing/log-linear capture-recapture math and `coverage/benchmark.py` are the one place that's genuinely analytical (aggregating across sources/companies), and that's the only place a `duckdb.sql("SELECT * FROM sqlite_scan('tracker.db','inbox')")`-style read-only attach would pay for itself — not the inbox/tracker hot path.

Risk flag: DuckDB's Python wheel is small and dependency-free (`duckdb` 1.5.4, win_amd64 wheel = 13.1MB, confirmed via PyPI — [pypi.org/project/duckdb](https://pypi.org/project/duckdb/)), so base bundling is fine, but its **extension-loading** path has an open, version-specific Windows+PyInstaller bug: loading the `motherduck` extension causes an access-violation segfault in a PyInstaller-frozen exe on DuckDB 1.5.0/1.5.1 (worked fine on 1.4.x) — [duckdb/duckdb#21602](https://github.com/duckdb/duckdb/issues/21602). Core (extension-free) DuckDB querying is not implicated, but any future use of `httpfs`/`sqlite_scanner` extensions inside the frozen exe needs to be smoke-tested on the actual PyInstaller build before shipping, not just `python gui.py`.

**Recommendation: no DuckDB today.** Revisit only if/when `coverage/benchmark.py`-style cross-run analytics genuinely need sub-second aggregation over a 100k+ row inbox that SQLite (with proper indexes) can't hit — and even then, attach it read-only against the existing `tracker.db`, don't fork the schema.

## Vector storage: what's actually pip-installable and PyInstaller-safe

For a distributable single-file/no-server exe, the field narrows fast:

| Option | Fit for JobScout | Notes |
|---|---|---|
| **NumPy brute-force** (store embeddings as BLOB/npy, cosine in Python/NumPy) | **Best default** | No new C-extension risk, numpy is a PyInstaller-friendly package that "bundles out of the box" per PyInstaller's own docs. Brute-force cosine over tens-of-thousands of vectors is sub-100ms on commodity hardware with NumPy's BLAS-backed matmul; the literature is explicit that brute force stays "fast enough" at "small-to-mid scale" and only 20-50x-slower-than-IVFFLAT becomes a real problem in the millions-of-vectors range ([oneuptime.com flat-index guide](https://oneuptime.com/blog/post/2026-01-30-vector-db-flat-index/view), [IVFFLAT practical guide](https://nidly.substack.com/p/cosine-similarity-meets-ivfflat-practical)). JobScout's realistic ceiling (100k inbox rows × one embedding each) is exactly this regime. |
| **sqlite-vec** (`pip install sqlite-vec`) | Good if you want the vectors *inside* `tracker.db` rather than a side NumPy file | Pure C, no dependencies, "runs anywhere SQLite runs" including Windows; it's the maintained successor to the now-deprecated `sqlite-vss` (which wrapped Faiss and was abandoned by its own author over integration pain) — [asg017/sqlite-vec](https://github.com/asg017/sqlite-vec), [sqlite.ai state-of-vector-search write-up](https://marcobambini.substack.com/p/the-state-of-vector-search-in-sqlite). Ships as a tiny Python wheel (~160KB per platform on PyPI — [pypi.org/project/sqlite-vec](https://pypi.org/project/sqlite-vec/)) loaded via `sqlite3`'s `enable_load_extension` + `sqlite_vec.load(db)`. Caveat: it's a brute-force virtual-table scanner itself (no ANN index), and vectors must live in a separate `vec0` virtual table joined back to `inbox` by rowid — some Windows extension-loading friction has been reported by users on older Python builds ([sqlite-vec#13](https://github.com/asg017/sqlite-vec/issues/13)), so pin a version and smoke-test the frozen exe. |
| **sqlite-vector** (`pip install sqliteai-vector`, by the same author as the Substack benchmark) | Interesting, newer, less proven | Stores vectors in ordinary tables (no virtual-table indirection), supports quantization + "preload into memory," and beat sqlite-vec in the one available benchmark (100k × 384-dim FLOAT32 on M1 Pro: insert 563ms vs 1179ms; full-scan query 56.65ms vs 67.84ms; quantized query 17.44ms; preloaded query 3.97ms — [marcobambini.substack.com](https://marcobambini.substack.com/p/the-state-of-vector-search-in-sqlite)). Only one benchmark source found (the maintainer's own), so treat these numbers as directional, not independently verified. Windows support is claimed but not battle-tested at JobScout's scale. |
| **FAISS-flat** | Not recommended | A library, not a database — no persistence, no metadata filtering, no SQL join back to `inbox` rows; you'd hand-roll everything sqlite-vec/NumPy already give you, for a much larger binary dependency (`faiss-cpu` wheels are tens of MB and pull in BLAS/OpenMP runtime DLLs that are a known PyInstaller pain point on Windows). |
| **LanceDB / Chroma** | Not recommended | Both are "embedded, no server" but are still standalone databases with their own on-disk format (Lance columnar format for LanceDB) living *outside* `tracker.db` — that breaks the "single SQLite file per project" model JobScout already has, complicates backup/移动/multi-project semantics (`workspace.py`'s per-project dirs), and neither buys anything JobScout needs below the millions-of-vectors range where LanceDB's disk-based IVF-PQ indexing starts to matter ([LanceDB vs Chroma comparison](https://zilliz.com/comparison/chroma-vs-lancedb)). |

**Recommendation:** if/when embeddings are added (e.g., semantic "find similar jobs" or duplicate-detection-by-meaning rather than by URL), start with **NumPy brute-force stored as a BLOB column on `inbox`** (`embedding BLOB`, `np.frombuffer(row["embedding"], dtype=np.float32)`) — zero new packaging surface beyond adding `numpy` to `requirements.txt`/`app.spec` hiddenimports (already proven safe by PyInstaller's own docs and by the fact `coverage/estimators.py` already lazy-imports it). Move to `sqlite-vec` only if profiling shows the Python-loop cosine pass is a real bottleneck (unlikely under 100k rows) or you want the vectors queryable via plain SQL joins.

## Schema design for a growing corpus

1. **Provenance / per-source raw capture.** `inbox` today collapses each source's fields into one flat row with no raw payload retained — once `match/scorer.py` or `match/facts.py` changes, there's no way to re-derive facts from the original API response. Add a lightweight `source_id TEXT` (source's native job id, distinct from `norm_url`) and consider a `raw_json TEXT DEFAULT ''` column (SQLite's `json1` extension is built-in, so `json_extract(raw_json,'$.foo')` works with no new dependency) populated only when cheap (most clients already parse a dict before building `JobResult`). This is the standard "keep the source-of-truth payload, derive everything else" ETL pattern — SQLite's TEXT-JSON storage (no JSONB dedup like Postgres, but adequate for read-mostly small payloads) fits fine at this scale.
2. **Dedup keys already correct, keep the pattern.** `models.normalize_url` + `inbox.norm_url UNIQUE` ([tracker/db.py:135](tracker/db.py:135)) is the right content-addressed-key pattern (canonicalize → hash/string → UNIQUE constraint) and should be the template for any new dedup surface (e.g., a future `companies` table keyed on `(ats_type, slug)` the way `save_companies` already dedups in Python — just enforce it as a real `UNIQUE(ats_type, slug)` constraint instead of an O(n) set scan).
3. **Incremental/delta ingestion.** `inbox_add_many`'s `INSERT OR IGNORE` + pre-computed `seen_urls()` set ([tracker/db.py:492-546](tracker/db.py:492)) is already a reasonable delta-ingestion pattern for the current scale. At 100k+ rows, pulling all `tracked_urls()`/`dismissed_urls()` into a Python `set()` every run (`tracker/db.py:461-487`) becomes the bottleneck before the SQLite writes do — switch that seen-check to a SQL `WHERE norm_url NOT IN (SELECT ...)`/`NOT EXISTS` anti-join, or at minimum index `dismissed.url` and `applications.url` (only `applications.status` and `inbox.company` are indexed today — [tracker/db.py:122](tracker/db.py:122), [tracker/db.py:159](tracker/db.py:159); there is no index on `applications.url` despite `tracked_urls()` scanning it every run).
4. **Retention/pruning.** No TTL/retention policy exists on `inbox` today beyond manual `inbox_delete`/`inbox_dismiss`. As the inbox grows to 100k+, add a scheduled prune (e.g., dismiss-and-archive rows older than N days with no user interaction) — SQLite handles `DELETE ... WHERE date_added < ?` fine, but do it in batches (a few thousand rows per transaction) to avoid a single huge WAL-mode transaction; SQLite's WAL docs specifically warn that transactions "in excess of a gigabyte" can fail with an I/O error and recommend keeping WAL transactions to a few dozen MB (relevant if a bulk-import or bulk-prune ever touches the whole table at once) — [sqlite.org/wal.html](https://sqlite.org/wal.html), corroborated by a real-world "20GB WAL" war story ([loke.dev](https://loke.dev/blog/sqlite-checkpoint-starvation-wal-growth)).
5. **History/versioning.** `status_history` and `score_history` already implement the right append-only audit-log pattern (snapshot-before-update, one row per change) — this is the standard SCD-Type-2-lite approach for a local SQLite app and should be the template for any future "job posting changed" tracking (e.g., if a scraper re-visits a board and a posting's salary/description changed, log it the same way rather than overwriting silently).
6. **Registry migration.** Once `companies.json` crosses roughly 5k-10k entries (an order of magnitude above today's 301), migrate it into a `companies` SQLite table (`id, name, ats_type, slug, industries_json, added_at`, `UNIQUE(ats_type, slug)`) inside a shared registry DB, keeping `companies.json` as an *import/export* format for user-editability and portability (matches the existing "user-wins, preserves comments/examples" contract in `save_companies` — [scrape/company_registry.py:179](scrape/company_registry.py:179)). This gets indexed lookups by `ats_type`/`industry` instead of a full JSON parse + Python filter on every call, and a real `UNIQUE` constraint instead of a hand-rolled set-membership check.

## FTS for search

Add a **contentless/external-content FTS5 table** over `inbox` (`title`, `company`, `location`, `description`) rather than a duplicate-content FTS table — external-content FTS5 keeps the index separate from the source data so `inbox.description` isn't stored twice, which matters once it's holding 5000-char JD snapshots for 100k rows ([SQLite FTS5 docs pattern](https://sqlite.work/optimizing-fts5-external-content-tables-and-vacuum-interactions/)). Bulk-populate via `INSERT INTO fts_inbox(rowid, title, ...) SELECT id, title, ... FROM inbox` (the fast bulk path per SQLite's own guidance), keep it in sync with triggers on `inbox` insert/delete, and run `INSERT INTO fts_inbox(fts_inbox) VALUES('optimize')` after any bulk import/prune rather than on every write (each `optimize` rewrites the whole index) — trigger it the same way `init_db()` already gates expensive work behind `PRAGMA user_version`, e.g. only when the row-count delta since the last optimize exceeds ~10%. This directly closes the gap noted above: there is currently no way for a user to search what's already been triaged into the inbox, which is a real usability problem for dad's non-technical workflow once his inbox has hundreds/thousands of rows.

## Keeping it fast as it grows

- Add the three missing pragmas to `get_conn()` ([tracker/db.py:55-63](tracker/db.py:55)): `PRAGMA synchronous=NORMAL` (safe under WAL, meaningful write-throughput win for `inbox_add_many`'s per-run batch inserts), `PRAGMA temp_store=MEMORY`, and a bounded `PRAGMA mmap_size=` for read-heavy GUI queries — all sourced from the well-cited SQLite scaling benchmark ([phiresky.github.io](https://phiresky.github.io/blog/2020/sqlite-performance-tuning/)), which demonstrated 100k SELECTs/sec at multi-GB scale with these four pragmas.
- Wrap `inbox_add_many`'s per-job inserts in a single explicit transaction for the whole batch (currently relies on the `with get_conn() as conn:` context manager's implicit commit-on-exit, which is likely already one transaction — verify, since Python's `sqlite3` autocommit-per-statement default can silently defeat this if `isolation_level` isn't what's assumed).
- Add `PRAGMA wal_checkpoint(TRUNCATE)` on clean app shutdown (already used once for the schema-migration backup path — [tracker/db.py:96-99](tracker/db.py:96) — but not on normal exit), so the `.db` file the user sees on disk reflects reality rather than growing a persistent WAL sidecar file, per the widely-repeated "TRUNCATE checkpoint before closing the last connection" desktop-app guidance ([PhotoStructure WAL guide](https://photostructure.com/coding/how-to-vacuum-sqlite/)).
- Periodic `PRAGMA incremental_vacuum` (requires `PRAGMA auto_vacuum=INCREMENTAL` set once at DB creation — this must be set before any tables exist, so it needs a version-gated migration path if retrofitted onto existing project DBs) to reclaim space from pruned/archived rows without the full-rewrite cost of `VACUUM`.
- Per-project DB files (`projects/<slug>/tracker.db`) are already a natural, good partitioning strategy — it caps how large any single SQLite file needs to get (each person/project's corpus, not one global 100k+ mega-file), and keeps queries fast without needing table partitioning tricks.

## Bottom line for JobScout

Keep SQLite as the single-file, no-server, stdlib-only store — that's the right call at every scale this app will hit and is non-negotiable given the free/local/distributable constraint. The concrete, cited, actionable work is: (1) finish the WAL pragma set (`synchronous=NORMAL` + `temp_store=MEMORY` + `mmap_size`) in `tracker/db.py:55-63`, S effort, no legal/packaging risk; (2) add an external-content FTS5 index over `inbox` for in-app search, S-M effort, none; (3) migrate `companies.json` to an indexed SQLite `companies` table once the registry meaningfully grows (keep JSON as import/export), M effort, none; (4) if/when semantic matching is wanted, use NumPy brute-force embeddings on an `inbox.embedding BLOB` column first, escalate to `sqlite-vec` only if profiling demands it — both are S-M effort, low risk, and both avoid DuckDB/LanceDB/FAISS's added packaging surface for a workload that never needs their scale.

## Key recommendations

- **[S/medium — safe write-throughput win for daily-run batch inserts and read-heavy GUI queries, zero behavior change for users/risk:none]** Add PRAGMA synchronous=NORMAL, temp_store=MEMORY, and a bounded mmap_size to tracker/db.py get_conn() (currently only sets WAL + busy_timeout)  
  
- **[S/low-medium — housekeeping/disk hygiene, matters more as inbox grows/risk:none]** Add a TRUNCATE wal_checkpoint on clean app shutdown so the visible .db file doesn't leave a growing WAL sidecar  
  
- **[M/high — closes a real usability gap (no way to search already-triaged postings today), matters most for dad's non-technical workflow at scale/risk:none]** Add an external-content FTS5 virtual table over inbox(title, company, location, description) with insert/delete triggers and a batched 'optimize' after bulk imports/prunes  
  
- **[M/medium — removes full-file JSON parse + Python-side dedup scan on every registry call as it scales toward the stated 10k-100k target/risk:none]** Migrate companies.json to an indexed SQLite companies table (UNIQUE(ats_type,slug)) once the registry grows past ~5k-10k entries; keep JSON as an import/export format to preserve user-editability  
  
- **[S/medium — avoids an O(n) full scan + Python set materialization on every daily run once tables reach tens of thousands of rows/risk:none]** Index applications.url and switch tracked_urls()/dismissed_urls() from Python-side set-building to a SQL anti-join (NOT EXISTS) as the inbox and tracker grow  
  
- **[S-M/medium — meets JobScout's realistic scale (thousands-to-low-hundred-thousands of vectors) with zero new packaging risk; numpy is already PyInstaller-proven and JobScout already soft-depends on it/risk:none]** For future semantic/embedding features, store embeddings as a BLOB column on inbox and do brute-force NumPy cosine similarity rather than adding a vector database  
  
- **[S-M/low today (no current need), medium if semantic search ships/risk:none]** If a query-in-SQL vector need emerges later, use sqlite-vec (pure C, ~160KB wheel) over sqlite-vss (deprecated), FAISS (no persistence/metadata), or LanceDB/Chroma (break the single-file-per-project model); pin the version and smoke-test extension loading in the actual PyInstaller build first  
  
- **[S (decision), L (if ever implemented)/low — avoids adding unneeded packaging risk for a workload SQLite already handles/risk:none]** Do not adopt DuckDB as a primary store; only consider attaching it read-only for coverage/benchmark.py-style cross-run analytics if SQLite aggregation genuinely becomes the bottleneck, and smoke-test any extension loading (httpfs/sqlite_scanner) in the frozen exe first given the documented Windows PyInstaller extension-loading bug (duckdb/duckdb#21602)
