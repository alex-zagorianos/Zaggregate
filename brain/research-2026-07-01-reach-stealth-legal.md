# Research — Stealth fetching (JS/anti-bot) + the legal line (2026-07-01)

_Confidence: n/a_

## Summary
JobScout's existing design — Scrapling's `StealthyFetcher` as a lazy, opt-in, own-registry-only fallback (`scrape/stealth_fetch.py`, gated by `config.SCRAPLING_FALLBACK`) — is already the right shape for 2025-2026 tooling and law. Technically, single-page-per-run stealth browsing (Scrapling/Patchright/Camoufox) beats TLS-impersonation libraries (curl_cffi) for JS-rendered SPAs but costs ~250-900MB RAM and a ~150-300MB one-time Chromium/Firefox download, which is why it must stay opt-in in a PyInstaller exe. Legally, post-Van Buren (2021) and hiQ v. LinkedIn (9th Cir. 2022, settled Dec 2022) and Meta v. Bright Data (N.D. Cal. 2024), the CFAA does not reach public-page scraping, but breach-of-contract/ToS and — critically for JobScout's EU users — GDPR liability (LinkedIn's own €310M DPC fine; KASPR's €240k CNIL fine specifically for scraping LinkedIn contact data) remain very live risks that track authentication state and personal-data content, not "did you use a headless browser." The rule that falls out: stealth-render your own low-volume company-career-page registry (public, non-authenticated, JobScout's own curated list); never stealth-render LinkedIn/Indeed/authenticated surfaces — that boundary is already implicit in JobScout's architecture (browser extension = user-gated capture for those) and should be made explicit and enforced with a same-domain-only + registry-only guard and a dedicated rate limiter, which currently doesn't exist for the stealth path.

## Findings
## 1. Tool landscape, 2025-2026 state of the art

| Tool | Approach | Bypasses | Overhead | Reliability/maintenance |
|---|---|---|---|---|
| **Scrapling `StealthyFetcher`** (JobScout's `scrape/stealth_fetch.py`) | High-level fetcher; as of Scrapling ≥0.3.13 the default engine moved from a patched **Camoufox** to **Patchright** (Camoufox stays available as an opt-in alt engine) | Cloudflare Turnstile/Interstitial (JS, invisible, managed variants) "automatically"; patches CDP/WebRTC leaks, isolates JS execution, adds canvas noise, defeats headless-mode + timezone-mismatch checks | ~150-300MB one-time browser download (JobScout's own code comment says "~300MB"); ~250-900MB RAM per page while rendering; seconds-to-~60s per fetch (Cloudflare challenges can take up to 60s) | Actively maintained (PyPI, docs at scrapling.readthedocs.io); a public 2026 benchmark put raw Scrapling at **58% bypass success / 29.16s avg**, well below specialized paid APIs — treat as "good enough for occasional own-registry JS shells," not an industrial unblocker |
| **Camoufox** (Firefox-based anti-detect browser) | Compiled Firefox fork with deep fingerprint spoofing (canvas, WebGL, audio, `navigator`, geoip-consistent headers) | With `geoip=True`, a March-2026 community benchmark reported **100% pass on Instagram/Reddit/X/LinkedIn** login-wall pages | Lighter than Chromium: ~200MB RAM vs Chrome's 800MB+, ~100-200MB binary | Actively maintained (github.com/daijro/camoufox); Scrapling still supports it as an engine option |
| **Patchright** | Drop-in Playwright fork that patches Chromium at the **CDP protocol level** (not just JS injection) — ships pre-patched Chromium binaries | Beats plain Playwright/playwright-stealth on CDP-runtime and basic fingerprint checks; still Chromium-only (no Firefox/WebKit) | Same order as vanilla Playwright Chromium (~280MB download, 250MB+ RAM/page) | Actively maintained (github.com/Kaliiiiiiiiii-Vinyzu/patchright); a 2026 benchmark (ianlpaterson.com) found it and other stealth wrappers **still hit a Cloudflare interstitial in some configs** that a plain-CDP Chrome 148 session passed — no tool is a permanent bypass |
| **playwright-stealth** | JS-injection patch layer on stock Playwright | Passes basic bot checks; explicitly weak vs Cloudflare/DataDome higher tiers because JS overrides are inherently detectable at runtime | Same as vanilla Playwright (no extra binary) | Lowest effort to add, but the Python port has seen **sporadic maintenance** — patches lag weeks-to-months behind detector updates |
| **undetected-chromedriver / nodriver** | Selenium-CDP patch (UC) → direct-CDP-no-Selenium-shim successor (nodriver) | Fine against basic/older protections; nodriver's direct-CDP approach is the more current pick for 2026 | Standard Chromium footprint | UC is effectively legacy; **nodriver** is the maintained successor |
| **curl_cffi / tls_client (JA3/JA4 impersonation)** | HTTP client binding to `curl-impersonate` (BoringSSL) or a Go `uTLS` wrapper — spoofs TLS/JA3/JA4 + HTTP/2 fingerprints, **no JS execution** | Solves the *TLS-fingerprint* detection layer only; does nothing for JS challenges, canvas/behavioral checks | Tiny — a compiled cffi wheel (a few MB), no browser process, ~150ms/request vs ~2000ms for a fresh browser context | Actively maintained (`lexiforest/curl_cffi` on GitHub, PyPI); **JA4+ is now the baseline** across Cloudflare/AWS/major WAFs in 2026, so pure TLS mimicry alone is decreasingly sufficient against sites that also layer JS+behavioral scoring |
| **Botasaurus** | Selenium-based "anti-detect ecosystem" — fingerprint rotation, human-like interaction timing, proxy/UA rotation, Chrome-extension support | Markets itself against Cloudflare/PerimeterX/DataDome at scale | Selenium/Chromium-class overhead, heavier framework surface than a single-purpose fetcher | Actively developed (`omkarcloud/botasaurus`); oriented at *scale* scraping operations, not a one-shot local fallback — overkill for JobScout's use case |
| **Cloudflare / DataDome / Turnstile reality check** | — | Cloudflare protects **~21% of all websites** (Jan 2026); plain out-of-the-box Playwright/Puppeteer success rates on protected sites have **fallen from ~95% to under 20% in the past year**; Cloudflare ships detection updates weekly, DataDome retrains continuously | — | **Any specific bypass claim is time-perishable** — this whole table should be re-verified every few months before hardening JobScout's reliance on it |

**Bottom line for JobScout:** Scrapling+Patchright/Camoufox is the right *category* of tool (already chosen) because it renders real JS, which pure TLS-impersonation (curl_cffi) cannot do, and JobScout's targets are JS-SPA career pages, not TLS-fingerprint-only blocks. curl_cffi is not a substitute here but could be a *cheap first escalation rung* before the heavy browser (see recommendations). Botasaurus/undetected-chromedriver bring no capability Scrapling lacks and add framework weight — skip them.

## 2. The legal line

**CFAA — narrowed to almost nothing for public pages.** *Van Buren v. United States*, 593 U.S. ___ (2021), held CFAA "exceeds authorized access" is about *gates-up-or-down* file/database access, not violating a use-policy. Applying that, the Ninth Circuit in *hiQ Labs, Inc. v. LinkedIn Corp.* (9th Cir. 2022) again held that **scraping data a site has made public, without login, is outside the CFAA's reach even after a cease-and-desist**. [EFF summary](https://www.eff.org/deeplinks/2022/04/scraping-public-websites-still-isnt-crime-court-appeals-declares), [White & Case](https://www.whitecase.com/insight-our-thinking/web-scraping-website-terms-and-cfaa-hiqs-preliminary-injunction-affirmed-again).

**But hiQ still lost on everything else.** The case didn't end in hiQ's favor — it *settled* in December 2022 after LinkedIn won summary judgment on **breach of contract** (hiQ had agreed to LinkedIn's User Agreement / created accounts and used crowdsourced fake profiles to get around blocks) plus **trespass to chattels** and **misappropriation** under California common law; the consent judgment includes a **$500,000 judgment against hiQ and a permanent injunction** ordering it to stop scraping and delete everything it collected. [Proskauer summary of the settlement](https://www.proskauer.com/blog/hiq-and-linkedins-data-scraping-battle-ends-with-proposed-judgment), [Morgan Lewis](https://www.morganlewis.com/blogs/sourcingatmorganlewis/2022/12/linkedin-v-hiq-landmark-data-scraping-suit-provides-guidance-to-data-scrapers-and-web-operators). **Takeaway: CFAA is dead as a threat for public-page scraping, but ToS-as-contract and common-law torts are not — and they turn on whether the scraper has/had an account or agreed to terms**, not on the tooling used.

**Meta v. Bright Data (N.D. Cal., Jan 2024) sharpened exactly that distinction.** Judge Chen granted summary judgment *for* Bright Data: Meta's ToS bind "your use" of the *logged-in account*, so **logged-out scraping of public Facebook/Instagram pages is not a breach of contract at all** — and a "survival clause" trying to bind a terminated account-holder forever was unenforceable. [Bright Data's writeup](https://brightdata.com/blog/web-data/court-rules-in-favor-of-bright-data-in-meta-v-bright-data-case), [Lowenstein Sandler analysis](https://www.lowenstein.com/news-insights/publications/client-alerts/meta-v-bright-data-ruling-has-important-implications-for-webscraping-activities-by-investment-advisers-im). **X Corp v. Bright Data** (N.D. Cal., May 2024) went further, dismissing X's ToS/state-law claims as **preempted by the Copyright Act** for scraping public tweets (on appeal to the 9th Cir.). [Skadden](https://www.skadden.com/insights/publications/2024/05/district-court-adopts-broad-view), [MoFo](https://www.mofo.com/resources/insights/240604-california-federal-court-holds-x-s-claims). **Net effect: logged-out, no-account, public-page scraping is now on firmer legal ground in the Ninth Circuit than it was even a few years ago** — but this line of cases is corporate-scale commercial resale litigation, not a template that makes *authenticated* scraping (logging into LinkedIn, scripting past its login wall) safe; that's precisely what hiQ lost on.

**robots.txt has essentially no binding legal force**, but is evidence of "good faith" if a dispute arises. No US statute requires obeying it; it isn't a contract. The *Ziff Davis v. OpenAI* (S.D.N.Y., 2025) court went further and held robots.txt directives aren't even a DMCA "technological protection measure" — comparing them to a "keep off the grass" sign. [Eric Goldman's Technology & Marketing Law Blog](https://blog.ericgoldman.org/archives/2025/12/are-robots-txt-instructions-legally-binding-ziff-davis-v-openai.htm), [robotstxt.org FAQ](https://www.robotstxt.org/faq/legal.html). JobScout's `discover/career_link.py` already reads `robots.txt` only to harvest the `Sitemap:` line (not as an allow/deny gate) — that's a reasonable, defensible-but-not-required signal to also *honor* an explicit `Disallow:` on a target path before stealth-escalating, cheap to add, and strengthens the good-faith posture.

**GDPR is the sleeper risk that CFAA/ToS analysis misses, and it hits precisely the sites Alex would be tempted to stealth-scrape.** Two 2024 EU enforcement actions are directly on point:
- Ireland's DPC fined **LinkedIn €310M** (Oct 2024) for using member data without a valid legal basis. [Euronews](https://www.euronews.com/next/2024/10/24/irish-watchdog-slaps-linkedin-with-310m-fine-for-gdpr-violations), [Goodwin](https://www.goodwinlaw.com/en/insights/publications/2024/11/insights-finance-dpc-irish-data-protection-commission-fines-linkedin).
- France's CNIL fined **KASPR €240,000** (Dec 2024) specifically for **scraping LinkedIn contact details**, including details users had masked. [CNIL](https://www.cnil.fr/en/data-scraping-kaspr-fined-eu240000).

The KASPR fine is the closest real-world analogue to "a tool that scrapes LinkedIn profile/contact data" and shows regulators pursue the *scraper*, not just the platform, when the target is **personal data of identifiable people** (names, emails, employment history) — irrespective of CFAA or ToS outcomes. This is a distinct, EU-law axis: even a technically-and-CFAA-legal public scrape of an EU resident's personal data can trigger GDPR liability for the party doing the scraping and processing.

**The core distinction to hold onto: technical unblocking ≠ legal permission.** Getting past Cloudflare/Turnstile is an engineering fact; whether you're allowed to be there is a separate question that turns on (a) authentication state (public vs. behind-login), (b) whose data it is (company info vs. an identifiable person's data), (c) volume/purpose (personal low-volume research use vs. commercial resale — hiQ/Bright Data were monetizing at scale), and (d) explicit contrary signals (an explicit robots.txt `Disallow` or a cease-and-desist raises risk even where CFAA doesn't apply).

## 3. Rule-set for JobScout

**Stealth fetch OK (keep/extend current design):**
- Target is a URL from JobScout's **own curated company registry** (`scrape/company_registry.py`) — i.e., a company's own public `/careers` page, not a third-party aggregator.
- Page is **public/unauthenticated**, contains company-authored job postings (often the same JSON-LD `JobPosting` schema `scrape/jsonld_scraper.py` already parses) — not personal data about individuals.
- Volume stays **low and per-user-driven** — one dad, one Alex, running the app occasionally, not a always-on crawler hitting thousands of URLs/hour. This matches JobScout's constraint set (free/local/legal) and is the opposite fact pattern from hiQ/Bright Data-scale commercial resale.
- The fetch is used only as an **escalation of last resort** after a plain `requests` 403/timeout or a detected JS-shell (`_looks_js_shell` in `scrape/direct_scraper.py:18`) — exactly the current gating logic.
- Feature stays **opt-in** (`config.SCRAPLING_FALLBACK`, GUI "Enable stealth fetching" menu item, `gui.py:2690`) so the ~300MB Chromium/Camoufox download and the RAM spike never surprise a non-technical user like Alex's dad on first run.

**Stealth fetch NEVER (hard-block, don't build toward this):**
- **LinkedIn, Indeed, or any site requiring login/session cookies** to see the target content. This is exactly the hiQ fact pattern (account-bound ToS + trespass/misappropriation) and, per the KASPR fine, carries an *independent* GDPR risk for EU-resident personal data regardless of the CFAA/ToS outcome. JobScout's documented approach — a **browser extension for user-gated capture** of a page the human already opened and is looking at — is the correct compliant substitute and should stay the only path to that data, never an automated stealth crawl.
- **Mass/bulk scraping of aggregator search-results pages** (Indeed job-search listing pages, LinkedIn Jobs search) at any volume — this is squarely "JobSpy-style" mass scraping the project constraints already rule out, and is the fact pattern (commercial-scale, non-account-holder-but-still-a-competitor-database) that produced the *Meta*/*X Corp* litigation in the first place, even though those defendants ultimately won on the specific ToS/CFAA theories — litigation risk and cost is not "free."
- Any page carrying an explicit `Disallow:` in `robots.txt` for the specific path being fetched, or a site that has issued a cease-and-desist to JobScout/Alex — not legally fatal per *Ziff Davis*/*hiQ* CFAA holding, but a needless reputational/legal-cost risk for a hobby-scale free tool; cheap to check and skip.

**Concrete, actionable gaps to close (tied to real files):**
1. **Add a same-domain/registry-only guard** in `scrape/stealth_fetch.py::fetch_html` — currently it will render *any* URL string that `direct_scraper.py` hands it. Add an assertion/allowlist check that the URL's host matches the `CompanyEntry` it was invoked for (already the only caller, but nothing stops future misuse routing an aggregator URL through it). Effort: S.
2. **Add a per-domain rate limiter for the stealth path.** Unlike the API clients (`search/http_util.RateLimiter`), `scrape/direct_scraper.py` and `scrape/stealth_fetch.py` have **no rate limiter at all** today (confirmed by grep — no `RateLimiter`/`sleep` in either file). A stealth browser fetch is the most detectable, most server-costly request JobScout makes; add a small per-host cooldown (e.g., reuse `search/http_util.RateLimiter`) before escalating to `StealthyFetcher.fetch`. Effort: S. Impact: keeps "low volume" true in practice, not just in policy, and reduces the odds of tripping a target's own bot-defense escalation.
3. **Honor an explicit `robots.txt Disallow`** for the specific path before stealth-escalating (reuse the existing robots fetch in `discover/career_link.py:70` `_sitemap_urls_from_robots`, add a sibling `_is_disallowed(url)` check). Not legally required post-*Ziff Davis*, but cheap goodwill and reduces takedown-request risk for a free hobby tool. Effort: S.
4. **Pin/track the Scrapling version and its default engine.** `requirements.txt:18` currently pins bare `scrapling` (unversioned) — Scrapling's default stealth engine changed from Camoufox to Patchright at 0.3.13; an unpinned upgrade could silently change JobScout's fingerprint/behavior/download footprint. Pin a floor version and note the engine in a comment. Effort: S.
5. **Do not extend stealth fetch to LinkedIn/Indeed even for "just Alex's own account, low volume."** The KASPR precedent shows GDPR liability attaches to *scraping personal data*, independent of volume or account status; keep that boundary categorical, not a judgment call per-session. Document this as a one-line rule near `config.SCRAPLING_FALLBACK` so a future session doesn't casually "just add LinkedIn to the registry."
6. **Optional (M effort, low priority): add curl_cffi as a cheap middle rung** between plain `requests` and the full `StealthyFetcher` browser in `direct_scraper.py`'s escalation ladder — for sites blocked purely on TLS/JA4 fingerprint (no JS challenge), curl_cffi avoids spinning up a browser at ~150ms vs ~2-5s, saving RAM/CPU on the low-power machines JobScout targets. Only worth it if telemetry shows a meaningful fraction of registry blocks are TLS-only, not JS-shell — verify before building.

Legal risk summary for the rule-set: **stealth-fetching JobScout's own public company-registry career pages = low legal risk** (no CFAA per Van Buren/hiQ-9th-Cir, no ToS privity since there's no account/login, no personal-data processing since it's company job postings). **Stealth-fetching LinkedIn/Indeed or any authenticated/personal-data surface = medium-to-high legal risk** (hiQ-style breach-of-contract/trespass exposure plus independent GDPR exposure per KASPR) and is explicitly out of scope for this feature.

## Key recommendations

- **[S/Closes a latent misuse path before it exists; keeps the feature legally scoped to 'own registry, public pages' by code, not just by convention./risk:low]** Add a same-host/registry-only allowlist check inside scrape/stealth_fetch.py::fetch_html so the stealth browser can only ever render a URL belonging to the CompanyEntry that triggered it (not just 'whatever direct_scraper.py currently happens to pass').  
  Currently the only caller is safe, but nothing in the function itself enforces the boundary that makes the legal analysis hold.
- **[S/Keeps JobScout's 'low volume, non-abusive' fact pattern true in practice, reduces chance of tripping a target's bot-defense escalation or generating a complaint./risk:low]** Add a per-domain RateLimiter (reuse search/http_util.RateLimiter) around the stealth-fetch escalation in scrape/direct_scraper.py; today there is no rate limiting at all on the stealth/direct-fetch path (unlike the API clients).  
  hiQ/Bright Data-style legal risk scales with volume and automation intensity; a rate-limited, occasional, single-user fetch looks categorically different from a crawler.
- **[S/Not legally required (Ziff Davis v. OpenAI, 2025) but cheap goodwill/good-faith signal and reduces takedown-request friction./risk:none]** Add an explicit robots.txt Disallow check (sibling helper to discover/career_link.py's _sitemap_urls_from_robots) and skip stealth-escalating a path the target has explicitly disallowed.  
  robots.txt has no binding force in US courts, but courts and site owners treat compliance as evidence of good faith.
- **[S/Prevents a silent engine/behavior/download-size change on a routine dependency upgrade./risk:none]** Pin scrapling to a known-good minor version in requirements.txt (currently unpinned) and document which stealth engine (Camoufox vs Patchright, default changed at 0.3.13) JobScout relies on.  
  Scrapling's default anti-bot engine changed under the hood between versions; unpinned installs could change fingerprint behavior without anyone noticing.
- **[S/Forecloses the highest-legal-risk feature request before it gets built; matches hiQ's actual loss condition (account/ToS privity + trespass) and the independent GDPR exposure shown by the KASPR €240k fine for scraping LinkedIn contact data./risk:high (if violated)]** Keep and document a hard, categorical rule: never stealth-fetch LinkedIn, Indeed, or any authenticated/session-gated surface, regardless of volume or whose account is used; keep the browser-extension user-gated capture as the only path to that data.  
  This is the one bright line in an otherwise fact-specific legal area; codify it so a future session doesn't casually add LinkedIn to the company registry's stealth-eligible set.
- **[M/Lower RAM/CPU/latency than spinning a browser for sites that block on TLS fingerprint alone; near-zero binary size impact for the exe./risk:low]** Optional: prototype curl_cffi (JA3/JA4 TLS impersonation) as a lightweight middle rung before the full browser in direct_scraper.py's escalation ladder, only if telemetry shows registry blocks are TLS-fingerprint-only (not JS-shell).  
  curl_cffi doesn't execute JS, so it can't replace StealthyFetcher for JS-SPA career pages, but it's much cheaper for pure-TLS blocks — verify the split exists before investing effort.
