## A. Headline problems (fix-first)

### A.1 — XSS chain: browser harvest → report (security)

- `report.html:84,94` — `job.url` injected into `href`; Jinja autoescape stops tag injection but **not** `javascript:`/`data:` schemes.
- `browser_receiver.py:27-33` — `Access-Control-Allow-Origin: *` on every response + `get_json(force=True)` → any site you visit can POST to `localhost:5002`.
- `browser_receiver.py:50-54` — POST elements not type-checked; a non-dict element 500s the whole report.
- **Fix:** validate URL scheme server-side in `_to_job_result` (reject non-`http(s)`); drop wildcard CORS (reflect extension origin only); `isinstance(j, dict)` guard.

### A.2 — Rate-limiter unreliable + no JSearch monthly accounting

- `adzuna_client.py:55-73` & twins — timestamp recorded _after_ request; no loop/evict, throttles only on single oldest slot.
- JSearch free tier = **200 req/month**; `JSEARCH_RATE_LIMIT=5` enforces only 5/rolling-60s, zero monthly tracking. Default run = 20 JSearch calls; ~10 runs exhausts the month silently.
- **Fix:** timestamp request-start, loop-and-evict; persistent monthly counter for JSearch.

### A.3 — Cache key omits `salary_min` → serves wrong stale results 24h

- `adzuna_client.py:109` & twins — `_cache_key(keyword, location, page)` ignores `salary_min`. Slug also collides (`"Cincinnati, OH"` == `"Cincinnati OH"`; `"controls/automation"` == `"controls automation"`).
- **Fix:** include `salary_min`; hash a normalized tuple instead of slugging.

### A.4 — Resume generator: no structured output + no prompt caching

- `generator.py:32-44` — JSON requested via prose; `_parse_response` strips one fence, breaks on prose/trailing text. **Fix:** tool-use with `input_schema` + `tool_choice` → read `content[0].input`.
- Static experience corpus re-sent uncached every call. **Fix:** `system` array + `cache_control: ephemeral`; posting only in user turn.
- `max_tokens=4096` can truncate into invalid JSON; bump ~8192, check `stop_reason`.

### A.5 — Salary parser duplicated + misfires

- Two regexes (`content.js:91` JS, `browser_receiver.py:124` Python) already diverge.
- `re.search(r"hr|hour", text)` substring-matches `hr` anywhere ("Pittsburgh", "Amherst", "HR Manager") → ×2080 multiplier on annual figures.
- **Fix:** one parser (Python); always send `salary_text`; anchor unit pattern to `/\s*(hr|hour)\b`.

## B. Structure

- **`CareersClient` breaks the abstraction** (`careers_client.py:33-52`) — `search()`/`parse_results()` are dead stubs; work in overridden `search_and_parse()`. Interface wrongly assumes one paginated dict call per source. Refactor: `JobAPIClient` (paginated) vs `CareersAggregator`, common `fetch()`.
- **Three near-identical API clients** — rate-limit + cache + deque copy-pasted; scrapers reinvent caching in `cache_helpers.py` (two cache impls). Extract shared `RateLimiter` + `Cache`. (`cache_helpers.write_cache` is atomic via `os.replace`; API clients' `_write_cache` is not — adopt it.)
- **GUI vs Flask duplication** — DB + resume pipeline shared (good); orchestration re-expressed twice (`resume/app.py:31-37` vs `gui.py:406-413`). Flask UIs now dead weight beside `gui.py`. Pick one front-end; keep Flask only for `browser_receiver`. Extract thin `services.py`.
- **Testability debt** — `sys.path.insert` everywhere; `config.py:22-23` mkdir at import; `print`-based logging (no levels).

## C. Efficiency

- **Search fully serial** (`search_engine.py:45-64`) — no parallelism across clients/keywords. Biggest latency win.
- **Career boards re-parsed per keyword** (`careers_client.py:54-70`) — ~200 parse passes where 20 fetches + one filter would do.
- Tracker 2-3 DB connects per refresh; `experience.md` re-parsed per call. Minor memoizations.

## D. Smaller correctness

Dedup key (`models.py:21`) too aggressive (collapses distinct roles) yet too weak cross-source (`""` vs `"Remote"`); MutationObserver re-harvests whole DOM + lost-update race (`content.js:174`); lexicographic date sort on heterogeneous formats (`search_engine.py:70`); no retry/backoff; `break`-on-error abandons later pages (`search_engine.py:57`); `--no-discover` help still says "DuckDuckGo" (`cli.py:141`); HTTPError handlers assume `e.response` non-None; salary fields trust API types (`salary_display` can raise on str); `(item.get("company") or {})` guards missing.

## E. Features / UX / completeness

1. **No cross-run / tracker dedup** — every search re-surfaces tracked/applied/dismissed jobs. No seen/dismissed state. #1 workflow gap.
2. **Search not in GUI** — must drop to terminal. Add a Search tab.
3. **No resume ↔ tracker link** — can't see which resume went where.
4. **Tracker missing job-hunt fields** — follow-up/next-action date, deadline, recruiter/contact, saved JD snapshot.
5. **No funnel analytics** — response rate, applied→phone→interview→offer, time-to-response, stale aging.
6. **No automation** — manual runs only; daily search → dedup → digest of new-only.
7. **Resume polish** — no ATS/keyword score, no preview, single variant.
