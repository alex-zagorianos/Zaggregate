# Research: Deduplication & Ghost/Stale-Job Quality Filtering (2026-07-01)

Scope: cross-source job dedup at aggregator scale + ghost/expired/stale-posting detection, benchmarked against JobScout's existing `coverage/entity.py` (job_key), `coverage/resolve.py` (rapidfuzz union-find, threshold 85), `models.normalize_url` (URL canon + Indeed `jk` collapse), and `match/ghost.py` (offline staleness heuristic).

---

## 1. Cross-source job deduplication

### 1.1 Exact-ID and URL canonicalization

Every major aggregator does URL/ID canonicalization as the first, cheapest dedup pass before any fuzzy matching — unwrap redirect/click-tracking wrappers, strip UTM/tracking params, collapse platform-specific job IDs (Indeed `jk`, Greenhouse `gh_jid`, Lever `lever-id`) to one canonical form. JobScout's `models.normalize_url` already does this (Google `/url`,`/aclk` unwrap, generic redirect-host/path regex, Indeed `jk` collapse, tracking-param strip while preserving identity params) — this matches industry practice and needs no structural change, only maintenance (new redirect-host patterns as sources are added).

### 1.2 Fuzzy text similarity: TF-IDF cosine vs. embeddings vs. character-overlap

- **TF-IDF + cosine similarity** is the traditional baseline for job-title/description dedup and still appears in current academic work — e.g. a 2026 IJERT study of job-aggregation systems built directly on "web scraping, APIs, TF-IDF, and cosine similarity" to catch repeated postings across sources ([IJERT](https://www.ijert.org/a-study-of-job-aggregation-systems-using-web-scraping-apis-tf-idf-and-cosine-similarity-ijertv15is050898)).
- The most rigorous applied study found — **"Combining Embeddings and Domain Knowledge for Job Posting Duplicate Detection"** (arXiv 2406.06257, in production use at a real job platform) — found that **no single method (character-overlap, embedding cosine, or weighted-keyword) is individually sufficient**; an ensemble ("Total Score" combining overlap-based Text/Skill Overlap Score + `distiluse-base-multilingual-cased-v1` sentence-embedding cosine similarity + inverse-frequency-weighted skill matching) reached **precision 0.89 / recall 0.99 / F1 0.94** on a hand-labeled 147-pair test set, vs. materially worse for any single method alone ([arXiv](https://arxiv.org/abs/2406.06257)). Key transferable finding: they used **posting-date proximity as a temporal blocking filter** before pairwise comparison, since dedup candidates are almost always posted within days of each other.
- A companion 2026 preprint, "LLM-Driven Evaluation of Text Embedding Similarities for Job Posting Deduplication" ([Preprints.org](https://www.preprints.org/manuscript/202506.1143)), pushes further into using LLM judges to validate embedding-similarity thresholds — relevant only if JobScout later wants an AI-adjudication tier for borderline dedup pairs (it already has an AI-ranking budget elsewhere in the pipeline that could be reused).
- **Assessment for JobScout**: `resolve.py`'s current approach (rapidfuzz `token_set_ratio` on title + `WRatio` on canonical company, weighted 0.6/0.4, threshold 85) is a lighter-weight cousin of the "overlap-based" leg of the winning ensemble above — reasonable for a free/local desktop app with no GPU embedding budget, but it is missing the **description-level signal** entirely, which is where the academic result gets its recall lift on paraphrased/re-templated reposts from different boards.

### 1.3 MinHash + LSH for near-duplicate detection at scale

- Kalibrr (a job platform, "deduplicating jobs from multiple sources") published a direct case study: LSH found postings "appearing over 200 times" in their corpus and ran **~4x faster than TF-IDF** at their scale, trading a small precision hit for throughput ([Kalibrr Research / Medium](https://medium.com/kalibrr-research/locality-sensitive-hashing-lsh-a-scalable-solution-for-deduplicating-jobs-from-multiple-sources-dd23460432de)) — the article doesn't publish shingle-size/band/row parameters, so treat it as directional evidence ("LSH scales better than pairwise TF-IDF") rather than a tunable recipe.
- Standard MinHash/SimHash LSH literature (Charikar simhash, Broder MinHash) is well covered by mature Python libs: `datasketch` (MinHash+LSH, pure Python, pip-installable, no compiled deps) and `simhash-py` / `python-simhash` (Charikar simhash + Hamming-distance near-dup clustering) — both used in large-scale LLM-training-data dedup pipelines as of 2026 ([Yury Kashnitsky walkthrough](https://yorko.github.io/2023/practical-near-dup-detection/), [seomoz/simhash-py](https://github.com/seomoz/simhash-py), [scrapinghub/python-simhash](https://github.com/scrapinghub/python-simhash)).
- **Assessment for JobScout**: at JobScout's actual scale (hundreds to low-thousands of jobs per run, not millions), MinHash/LSH's advantage — sub-quadratic candidate generation for O(n²)-prohibitive corpora — **does not pay for itself**. `resolve.py`'s blocking-key + O(n²)-within-block union-find is already the right shape at this scale (LSH's own literature frames it as a scale-up path once pairwise comparison stops being cheap, which is not JobScout's regime). Revisit only if a single run's candidate pool exceeds ~20-50K rows.

### 1.4 Blocking keys

`resolve.py._block_key` (canonical company, SOC code, location token) is a textbook blocking key — it's the same shape recommended by the general record-linkage literature (`dedupe`, Python Record Linkage Toolkit, Splink) to avoid the O(n²) pairwise blowup by only comparing within-block candidates ([dedupeio/dedupe](https://github.com/dedupeio/dedupe), [Practical Business Python record-linking survey](https://pbpython.com/record-linking.html)). One gap: blocking on SOC code is only as good as `entity.normalize_title`'s O*NET match; a title that falls through to the `00-0000`/confidence-0.0 fallback silently blocks with every other unmapped title system-wide, which could either over- or under-merge depending on volume in that bucket — worth a `coverage_industry`-style metric on fallback-rate.

### 1.5 Entity resolution for company-name variants

JobScout already uses `cleanco` (`coverage/entity.canonicalize_company`) for legal-suffix stripping, layered with `company_aliases.json` for hand-curated variants (e.g. Optum/UnitedHealth) — this matches the standard recommended stack: `cleanco` for suffix/legal-form normalization + `rapidfuzz.WRatio` for residual fuzzy matching, with a WRatio threshold around 90 commonly cited as the sweet spot where "unrelated pairs stay out while suffix/spacing variants merge" ([Practical Business Python](https://pbpython.com/record-linking.html)). JobScout's `_pair_matches` in `resolve.py` uses WRatio at a _blended_ 85 combined score (not isolated to company), which is slightly looser than the standalone-company 90 recommendation — acceptable since it's blended with title similarity, but worth a note if false-merges show up (subsidiary vs. parent, "Optum" absorbing an unrelated smaller "Optum Labs" spinoff, etc.).

- Heavier alternative for future consideration: `dedupe` (ML-trained blocking + pairwise classifier) or `Splink` (Fellegi-Sunter probabilistic linkage, scales via DuckDB) — both are overkill for JobScout's per-run volume and require either labeled training pairs (`dedupe`) or a Spark/DuckDB backend (`Splink`); not recommended now.

## 2. Ghost / expired / stale-job detection

### 2.1 schema.org `validThrough` — a concrete, currently-unused signal

Google's own JobPosting guidance is explicit and enforceable: publishers must either (a) keep `validThrough` populated and in the future for active postings, (b) set it to a past date once expired, or (c) 404/410 the page — sites that leave expired `JobPosting` markup live get a **Search Console "structured data policy violation"** warning and a reduced Google Jobs "trust score" over repeated offenses ([Google Search Central](https://developers.google.com/search/docs/appearance/structured-data/job-posting)). This means **legitimate company career sites using JSON-LD have a real economic incentive to keep `validThrough` accurate** — it is a stronger, source-attested freshness signal than any heuristic JobScout can derive from prose.

- **JobScout already parses this exact JSON-LD blob and throws the field away.** `scrape/jsonld_scraper.py:112` extracts `created = obj.get("datePosted")` but never reads `obj.get("validThrough")`, even though it's sitting in the same parsed object. This is the single cheapest, highest-confidence addition available: capture `validThrough` wherever JSON-LD is the source, and treat `validThrough < today` as a near-certain "expired, should not be in the inbox" signal rather than a soft ghost-score bump — it's a publisher-declared fact, not an inference. (API sources like the JSearch/Adzuna-style aggregator feeds JobScout also uses won't carry this field — it's JSON-LD-only.)

### 2.2 Ghost-job prevalence — 2024-2026 numbers (methodology varies a lot; cite the method, not just the %)

Numbers cluster in the 15-30% range for "listing behavior" studies and are noticeably higher for employer self-report surveys. Treat every number below as **methodology-dependent** — worth stating that caveat in-app if JobScout ever surfaces a "X% of postings may be ghost jobs" claim:

- **Clarify Capital** (175,000+ US listings analyzed, published via Forbes Apr 2026): ~1 in 7 postings (about 14%) remain active >30 days; worst industries wholesale (51%), mining (48%), biotech (43%); senior-level candidates hit hardest (21% encounter a ghost posting) ([Forbes, Apr 2026](https://www.forbes.com/sites/rachelwells/2026/04/09/1-in-7-job-postings-are-ghost-jobs-new-study-reveals-here-are-3-steps-to-avoid-fake-job-ads/)). Note this is a _different_ Clarify Capital output than the ~33% "employer self-report intent" figure cited elsewhere — same org, two different methodologies (listing-duration analysis vs. survey), do not conflate the two.
- **Greenhouse (ATS vendor, outcome data)**: 18-22% of postings on the platform show ghost-like patterns.
- **ResumeUp.AI (LinkedIn analysis)**: ~27.4% of active US LinkedIn postings classified ghost, defined as "still live past the average fill-cycle length for that role type."
- **LiveCareer** (918 HR professionals surveyed, Mar 2025): 45% "regularly" post ghost jobs, 48% more "occasionally" → 93% combined admission rate (self-report, upper bound).
- **Columbia Law Review analysis of BLS JOLTS data** (per a synthesis piece): government roles ~60% ghost-rate, education/health services ~50%, info/tech ~48% — these are the highest-cited industry breakdowns and come from a labor-economics reading of federal openings-vs-hires data rather than a job-board crawl, so they're structurally different from the platform-level numbers above.
- **Stack Overflow's own Dec-2024 survey-based writeup**: ~40% of hiring managers admit their company posted a ghost job in the past year; 85% of companies that do still interview candidates for the fake req; ~50% keep postings live because they're "always open" to strong candidates ([Stack Overflow Blog](https://stackoverflow.blog/2024/12/26/the-ghost-jobs-haunting-your-career-search/)).
- Congressional Research Service has a short explainer (IF12977) on ghost postings as a policy topic, confirming this is now a live legislative issue, not just a media narrative ([Congress.gov CRS IF12977](https://www.congress.gov/crs-product/IF12977) — fetch blocked by 403 in this session, cite as a pointer for follow-up, not content already verified here).

### 2.3 2025-2026 state legislation — relevant to JobScout's "surface a warning" framing

Multiple states are actively legislating ghost-job disclosure as of mid-2026, which validates ghost-job filtering as a real, not fringe, user need — and gives JobScout forward-looking language to borrow for in-app copy:

- **New York** Senate passed S8877 (Apr 28, 2026) requiring employers to disclose expected hiring timeline or explicitly state "collecting resumes for future opportunities" ([NY State of Politics](https://nystateofpolitics.com/state-of-politics/new-york/politics/2026/04/30/n-y--state-senate-passes-bill-to-crack-down-on--ghost-job--postings)).
- **Pennsylvania** "Ghost Job Postings Prevention Act" (introduced Mar 2026): requires stating existing-vs-anticipated vacancy, a hiring timeline, disclosure of how many times reposted in the prior year, and removal within 2 weeks of fill ([Forbes, May 2026](https://www.forbes.com/sites/michelletravis/2026/05/26/ghost-job-ads-are-latest-employer-tactic-targeted-by-state-lawmakers/)).
- **California** (Mar 2025) and **Kentucky** (Jan 2025) bills with similar vacancy-disclosure requirements; a federal **Truth in Job Advertising and Accountability Act (TJAAA)** was announced June 2025.
- Practical implication: some future postings may start carrying explicit "anticipated vacancy" / repost-count disclosures in-page — if/when that happens it becomes a much stronger structured signal than anything inferred today; worth a watch-item, not an action-item now.

### 2.4 Repost/re-list heuristics (industry practice, not formal research)

- LinkedIn/Indeed reposting is **employer-initiated, not automatic** — a "Reposted Xd ago" badge means the employer manually refreshed the listing to regain "new" ranking/algorithmic boost, while the underlying role may have been open for months ([HideJobs guide](https://hidejobs.com/learn/linkedin-reposted-jobs)). Practically: **posting-age computed from a single `datePosted` value is exactly the field platforms game via repost** — this is the strongest argument for weighting `validThrough` (§2.1) over `datePosted`-only age when both are available, since `validThrough` is not reset by a cosmetic repost the way `datePosted` typically is.
- No source found a rigorous published threshold for "reposted N times in M weeks = ghost"; it's treated as a soft signal industry-wide, consistent with JobScout's own explicit design choice to abstain on repost-count "since freshness history isn't retained yet" (`match/ghost.py` docstring). This is a real gap, not a false negative — if JobScout starts persisting `job_key` sightings across daily runs (which `is_new`/baseline tracking already partially does), a repost-count signal becomes cheap to add (see recommendations).

### 2.5 ML approaches to ghost/fraud detection

Academic ML work here targets **fraudulent** postings (scam/phishing, the EMSCAD-style Kaggle "Fake Job Postings" dataset) more than **ghost** (real company, no real intent) postings — the two problems are related but not identical, and JobScout's `ghost.py` docstring already correctly separates them (staleness ≠ fraud, kept out of the match score). Representative results: Gradient Boosting on empirical rule-set + POS-tag + bag-of-words features reached F1 0.88; Random Forest reached 96.4% accuracy on a similar fraud-classification task ([Springer](https://link.springer.com/article/10.1007/s00146-022-01469-0), [ACM/Neural Processing Letters](https://dl.acm.org/doi/abs/10.1007/s11063-021-10727-z)). These require labeled training data JobScout doesn't have and target a different problem (fraud, not evergreen-pipeline staleness) — **not recommended** as a build target; JobScout's rule-based `ghost.py` is the right-shaped tool for the staleness problem it's actually solving.
