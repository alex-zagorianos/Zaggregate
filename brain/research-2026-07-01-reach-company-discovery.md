# Research: Company/ATS-Board Discovery at Scale (Free + Legal)

**Date:** 2026-07-01
**Question:** Every free+legal way to enumerate employers and their ATS boards at 10k-100k+ scale, to grow JobScout's company registry (the real reach constraint per Session 24 — a req in Greenhouse/Workday/Lever auto-syndicates to LinkedIn/Indeed/Google, so registry breadth = reach without ever scraping those aggregator sites).
**Existing JobScout code:** `discover/cc_harvest.py` (Common Crawl CDX, per-host + host/domain-level), `discover/dataset_seed.py` + `seed_companies.py` (bulk import through a probe-verify gate), `discover/enumerate.py` (LLM metro enumeration), `verify_and_add.py`.

## 1. Common Crawl (already the generic-denominator winner — extend, don't replace)

**CDX API** (`index.commoncrawl.org`) — what `cc_harvest.py` already does. Free, no auth, no AWS account. Two query shapes:

- `url=host/*` — single host, e.g. `boards.greenhouse.io/*`.
- `url=<registered-domain>&matchType=domain` — the whole domain **and every subdomain/tenant**, e.g. `myworkdayjobs.com` returns every `<tenant>.wd5.myworkdayjobs.com` board in one paginated query. This is the completeness lever `harvest_host_index()` already uses (`page`/`pageSize`/`showNumPages`). [Common Crawl Index Server](https://index.commoncrawl.org/) · [pywb CDX API reference](https://github.com/webrecorder/pywb) · [cdx_toolkit](https://github.com/commoncrawl/cdx_toolkit) (a maintained Python CLI/lib wrapping CDX+Wayback if the hand-rolled fetch ever needs richer paging/retry logic).
- **Gotcha found in the repo:** `cc_harvest.py` hardcodes `_CDX_INDEX = ".../CC-MAIN-2025-05-index"` as the fallback. Common Crawl is now on **CC-MAIN-2026-25** (June 2026) — 2025-05 is ~17 months and ~9 crawls stale, meaning boards created since then are invisible unless a caller passes `crawl_id` explicitly. CC publishes a self-discovery endpoint, `https://index.commoncrawl.org/collinfo.json`, listing every available crawl id (`id`, `cdx-api`, `from`/`to`) in reverse-chronological order — cheap fix: fetch `collinfo.json[0].id` at harvest time instead of hardcoding, or at minimum bump the constant each session. [collinfo.json](https://index.commoncrawl.org/collinfo.json)

**Columnar (Parquet) Index** — the bulk-analytics sibling of CDX, queryable via AWS Athena, DuckDB, Spark, or Pandas/Polars against `s3://commoncrawl/cc-index/table/cc-main/warc/`. [Columnar Index docs](https://commoncrawl.org/columnar-index) · [cc-index-table](https://github.com/commoncrawl/cc-index-table) · [worked SQL examples](https://skeptric.com/common-crawl-index-athena/). Query shape:

```sql
SELECT url, url_host_name FROM "ccindex"."ccindex"
WHERE crawl = 'CC-MAIN-2026-25' AND subset = 'warc'
  AND url_host_registered_domain = 'myworkdayjobs.com'
  AND url_host_name LIKE '%.myworkdayjobs.com'
```

**Trade-off vs CDX:** the columnar index is partitioned only by `crawl`/`subset`, not by domain, so a single-domain query still scans the whole crawl partition — narrow `SELECT`s hit as little as 10MB (near-free) but broad ones can scan ~~150GB (~~$0.75 at Athena's $5/TB, no free tier). It pays off only when querying **many** ATS domains at once (single `WHERE url_host_registered_domain IN (...)` pass across greenhouse.io/lever.co/myworkdayjobs.com/ashbyhq.com/smartrecruiters.com/workable.com/recruitee.com/etc. in one scan) — cheaper than N serial CDX calls once the target-host list grows past ~dozens. For JobScout's current ~10-host list, CDX stays the $0 right tool; Athena becomes worth wiring only if `harvest_host_index`'s host list grows substantially. [Athena pricing](https://aws.amazon.com/athena/pricing/)

**Verdict:** effort LOW (already built), impact HIGH, legal risk NONE (public dataset, no ToS violated — you're reading a crawl archive, not scraping the ATS live). Action: (1) fix the stale `2025-05` default via `collinfo.json`, (2) leave CDX as the workhorse, (3) defer Athena until the host list is large enough to justify AWS billing setup.

## 3. GitHub awesome-lists / Google-for-Jobs / sitemap harvesting

- _*awesome-* curated lists_* are low-yield for scale (hundreds, not thousands) but high-precision and near-zero effort to mine once: [awesome-job-boards](https://github.com/tramcar/awesome-job-boards), [awesome-career-pages](https://github.com/CSwala/awesome-career-pages), [awesome-ats](https://github.com/bormaxi8080/awesome-ats). Treat as a P0 seed layer, not a growth engine.
- **Google for Jobs / schema.org JobPosting**: no longer a scale-discovery lever for JobScout. As of 2025 Google gated the Indexing API behind approval and restricted it to the posting company itself submitting its own pages ([Cavuno: Indexing API guide](https://cavuno.com/blog/google-indexing-api-job-postings), [dstribute: 2025 shake-up](https://dstribute.io/job-boards/google-jobs-shake-up-2025-navigating-the-new-indexing-api-restrictions/)) — there is no bulk "list every JobPosting Google has indexed" API for third parties. The _pattern_ is still useful passively: any company career page already has `JobPosting` JSON-LD (that's what feeds Google for Jobs), so JobScout's own per-company scraper should already prefer parsing that JSON-LD over HTML scraping — worth confirming `scrape/*` does this, but it's a per-company extraction detail, not a discovery-at-scale lever.
- **Sitemap.xml/robots.txt harvesting**: legal, free, but only scales per-company (you need the domain first). Best used as the **verification** step after a company/domain is already known (crawl `robots.txt` → `Sitemap:` lines → find `/careers` or `/jobs` sitemap) rather than a discovery-at-scale source on its own — this is effectively what `discover/career_link.py`'s resolver already targets, and the Session-23 finding that it only hits ~1/10 (JS-SPA boards) confirms sitemap/robots parsing alone under-performs vs. direct ATS-slug datasets.

**Verdict:** awesome-lists = LOW effort/LOW impact/NONE risk (do it, cheap). Google Indexing API = not viable for third-party bulk discovery in 2025-2026. Sitemap harvesting = keep as verification, not discovery.

## 4. Company registries (name → domain resolution, for turning "a company exists" into "here's its board")

- **SEC EDGAR**: `https://www.sec.gov/files/company_tickers.json` (and `_exchange.json`) — free, no auth, official, nightly-refreshed bulk ZIP of every CIK/ticker/name. [SEC EDGAR APIs](https://www.sec.gov/search-filings/edgar-application-programming-interfaces). Covers **public companies only** (~10-13k US issuers) — small slice, but perfect precision and free; good P0/priority layer (large/known employers = high job-posting cadence).
- **Wikidata SPARQL** (`query.wikidata.org`): free, no auth, queryable for `P856` (official website) joined against `P452`/`P31` (industry/instance-of-business) — good for building a name→domain map for mid-size and international companies EDGAR misses. Query construction is the effort cost (SPARQL learning curve), not access. [Wikidata Query Service](https://query.wikidata.org/) · [worked example](https://dev.to/minchulkim87/company-data-using-wikidata-n19)
- **OpenCorporates**: free API access exists but is licensed only to "Permitted Users" (individuals, journalists/NGOs/academics for public-benefit research) — **explicitly excludes commercial/corporate use** in its free/share-alike tier, and bulk data requires a signed data-transfer contract. [OpenCorporates API terms](https://opencorporates.com/terms-of-use-2/) · [licence](https://opencorporates.com/legal/licence). Since JobScout is free-distributed (not sold) this may qualify, but it's a gray area worth a manual read of the current ToS before wiring in — **do not scrape opencorporates.com directly**, that's explicitly prohibited.
- **State Secretary-of-State registries**: no federal consolidation exists; each state differs wildly. Some (Ohio) publish free monthly bulk files; most charge $500-$12k+/yr for bulk/UCC data, and Delaware (the dominant incorporation state) offers **no bulk/API access at all**. [OpenCorporates: sourcing directly from state registries](https://blog.opencorporates.com/2025/09/15/sourcing-data-directly-from-us-state-registries/). **Verdict: not worth pursuing** — cost/effort far exceeds yield versus the datasets in §2.
